# RegioBuild

**Проект для сравнения региональных строительных нормативов (РНГП) в России** —
с опорой на федеральный фон (СП 42.13330.2016 и точечные выдержки 123-ФЗ / СанПиН).

Telegram здесь — демонстрационный клиент. Ядро — **RAG + LLM-агент с обязательной
привязкой факта к пункту НПА**. API можно поднять на своей инфраструктуре и
встроить в другой продукт (CRM застройщика, LegalTech-кабинет, внутренний
портал юрслужбы) — без привязки к мессенджеру.

## Зачем это вообще

Бизнес выходит в новый субъект РФ — и сразу упирается в **правовую
фрагментацию**: у каждого региона свой акт о нормативах градостроительного
проектирования, своя плотность регулирования, свои «дыры». Где-то по автомойке
есть жёсткая норма машино-мест, где-то регион молчит и остаётся опираться на
федеральный каркас. Юристу и проектировщику это знакомо: часы ручной сверки
постановлений и приказов, риск пропустить существенное условие размещения.

В США на похожей «межштатной» фрагментации уже вырос целый слой LegalTech.
У нас вместо штатов — субъекты федерации, а удобного инструмента сравнения
РНГП до сих пор почти нет. **RegioBuild как раз про это:** не «поболтать с
ИИ про стройку», а получить **проверяемую справку** с номером пункта и
уровнем регулирования (регион / федерация).

### Узкий scope — осознанный MVP

В индекс взяты **пять регионов** (МО, Краснодарский край, Свердловская и
Новосибирская области, Татарстан) и **конкретный федеральный контур**
(СП 42 + curated по 123-ФЗ / СанПиН) под волну коммерческого размещения:
автомойка, склад, ТЦ, АЗС, офис, общепит, гостиница, медцентр и т.п.

Это не «весь классификатор Минстроя» и не полный свод всех СП РФ. Я сознательно
собрал **живой продукт**, который уже экономит время на типовых вопросах
размещения. Масштабирование — следующий слой: больше субъектов, ПЗЗ,
отраслевые своды, B2B-кабинет. Архитектура (отдельный API, векторный индекс,
провайдер LLM за абстракцией) к этому готова.

### Кому и сколько это экономит

| Кому | Что снимает с плеч |
|------|--------------------|
| **Юрист / in-house** | Первичный проход по РНГП и федеральному фону вместо часов ручного поиска пунктов |
| **Проектировщик / застройщик** | Быстрый ориентир по парковкам, СЗЗ, пожарке до глубокой проработки |
| **Бизнес любого масштаба** | Меньше простоев на «разберёмся с нормативом», меньше сюрпризов при выходе в новый регион |

Да, финальная ответственность всё равно на человеке и первоисточнике — так и
должно быть в LegalTech. Но **черновая сверка**, которая раньше жгла календарь
и бюджет, здесь ужимается до минут. Для сети объектов или экспансии в несколько
субъектов это уже не «удобный бот», а **операционная экономия**.

## Что умеет

1. **Требования в одном регионе** — обзор по типу объекта капитального
   строительства с разбивкой региональный / федеральный уровень.
2. **Сравнение двух регионов** — различия и совпадения, с цитатами пунктов.

Каждый существенный тезис в ответе должен опираться на retrieved-фрагмент:
есть **программный citation grounding**. Нет опоры — лучше честный отказ,
чем галлюцинация «от себя».

## Интеграция и развёртывание

RegioBuild заточен под то, чтобы его **не держали заложником одного хостинга**:

- **API-first:** FastAPI (`/info`, `/compare`, `/health`, `/metrics`) — бот
  всего лишь клиент. Тот же контракт можно дернуть из веб-UI, мобилки или
  чужого бэкенда.
- **Docker:** один образ, роль `SERVICE_ROLE=api|bot`; локально — compose.
- **Своя БД:** SQLite для пилота или Postgres + Alembic.
- **LLM за абстракцией `LLMProvider`:** в проде — GigaChat; интерфейс не
  прибит гвоздями к одному вендору.

Если рассматриваете покупку / white-label / встраивание в свой стек —
пишите в LinkedIn (контакты внизу). Репозиторий показывает, что продукт уже
не «ноутбук с экспериментом», а собранный контур.

Архитектура графа, ingestion и runtime: **[`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md)**.

## Стек

| Слой | Что стоит |
|------|-----------|
| Язык | Python 3.11, type hints, asyncio |
| Retrieval / NLP | `sentence-transformers` (multilingual MiniLM), Chroma |
| Классификация категорий | scikit-learn (TF-IDF + LogisticRegression) |
| Оркестрация | LangGraph |
| LLM | GigaChat API (тариф **Pro** в проде), абстракция `LLMProvider` (+ YandexGPT в коде) |
| Backend | FastAPI, Prometheus `/metrics`, Sentry |
| Данные | SQLAlchemy + Alembic (SQLite / Postgres), disk LLM-cache |
| Клиент | aiogram 3.x |
| Качество | pytest, eval Recall@k / MRR, матрица coverage волны 1 |
| Infra | Docker / docker-compose, CI (GitHub Actions) |

## С чем пришлось разбираться (end-to-end)

LegalTech на региональных НПА — это не «кинул PDF в ChatGPT». Ниже — не
жалобы, а реальные узлы, которые пришлось закрывать руками по всему циклу.

**Данные и ingestion.** Разные форматы актов, грязный HTML/таблицы, ложная
нумерация пунктов (`section_number="1"` из колонок таблицы), неравномерная
плотность корпусов по субъектам. Без антиjunk-парсера и ручной актуализации
retrieval тащит мусор, а юрист видит «п. 1» вместо нормы.

**Retrieval и grounding.** Канцелярит НПА ≠ язык пользователя; «автомойка» ≠
«автомоек» в таблице. Нужны stems, boost, фильтр usable-чанков и жёсткая
сверка citation с найденным текстом — иначе модель уверенно галлюцинирует
номера статей.

**Продукт и UX.** Длинные естественные фразы, опечатки, оффтоп и попытки
вытянуть секреты; лимиты Telegram на длину сообщения; честный дисклеймер
вместо «ИИ всё знает».

**MLOps / прод.** Тяжёлый embedder на ограниченной RAM, cold start, раздельные
роли api/bot в одном образе, квота LLM, cache, метрики, миграции audit-полей.
Пилот крутится на **Bothost Pro** (4 vCPU / 2 GB RAM / 15 GB) + **GigaChat Pro** —
это уже не toy-тариф «на поиграться», а осознанный прод-контур под demo и
нагрузку пилота.

Итого: пришлось быть и юристом по предметке, и data-инженером по корпусу, и
ML-инженером по retrieval/eval, и бэкендером по API, и девопсом по выкладке.
Так и задумывалось — **полный цикл продукта**, а не одна красивая демка.

## Запуск локально

```bash
python -m venv venv
venv\Scripts\activate          # Linux/Mac: source venv/bin/activate
pip install -r requirements.txt
copy .env.example .env         # Linux/Mac: cp .env.example .env
```

Ключи LLM и токен бота — в `.env`, затем:

```bash
alembic upgrade head
python -m app.ingestion.pipeline        # при необходимости пересобрать корпус
python -m app.embeddings.build_index
python -m app.classifier.train
python -m scripts.ingest_curated

uvicorn app.api.main:app --reload
python -m app.bot.main                  # второй терминал
```

Или: `docker compose up --build` (api + bot). Postgres: см.
`docker-compose.postgres.yml`.

## Тесты и качество

На тесты и eval я не жалел времени: без этого LegalTech быстро превращается
в генератор красивого бреда.

```bash
pytest                                 # unit / API / grounding / coverage волны 1
python -m app.eval.retrieval_eval      # Recall@k, MRR
python -m app.eval.answer_eval         # регрессия ответов агента
python -m scripts.audit_corpus         # junk ratio + coverage объектов
```

CI (GitHub Actions) гоняет лёгкий контур без torch/chroma на каждом пуше в
`main`. Для прода — smoke API после выкладки (`scripts/smoke_wave1_prod.py`).
